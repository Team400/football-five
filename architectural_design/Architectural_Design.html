Security

  • Our design addresses security with a focus on authentication and authorization. To ensure each user is given access to the proper information (which will be confirmed through access control lists), we intend to design our authentication and authorization based on a login system that will allow for a second factor of authentication (offered through phone numbers or federated identity). The two-factor authentication will protect users from their credentials being used in some malice way by an intruder. The two-factor authentication will also protect the system from possible brute force attacks and session hijacking attacks. With the login component at the user interface management level and the authentication & authorization component at the integrated services, the two can work together to ensure users are protected. Another way to keep our users safe will be through encryption of traffic at an application level which will also protect against session hijacking attacks.

    • Cross-cutting concerns

  • Cross-cutting concerns are addressed in our design based on each aspect of the layers having some responsibility in maintaining an acceptable level of security through authentication and authorization (within the user interface management & integrated services layers), validation and management of queries (within the user interface management & basic services layers), and input validation (within the user interface layer). Performance and reliability can then be improved because each layer has some responsibility for the upkeep of FantasyMax.

    • Seperations of Concerns

  • We address separation of concerns by using components that each do one thing and one thing only, along with a layered architecture, with layers of components of related functionality. For example, each component in our Application Services layer provides a single element of functionality that is critical to the overall function of the application. Together, these components provide the essential services of the application, while being structured to make them as independent as possible. In addition, none of the components in any layer is related to implementation, in adherence to the principle of separating the “what” from the “how.”

    • Interface Stability

  • We will focus on using stable interfaces, i.e., interfaces that change slowly or not at all, to reduce or eliminate the need for changing components that use these interfaces.

    • Avoid Duplicate Functionality

  • We want to avoid duplication of functionality because duplication can cause many problems down the line. Duplications makes it harder to refactor your code. We will limit components that provide similar services in our implementation. Our design will not have components that rely on the implementation of other components, this lack of implementation dependencies means that if the implementation of a component is changed during continuous integration, this change won’t require the other components that interface with it to change as well. All components will interact with each other through the API

    • Technology Choices

  • Database
    FantasyMax will be built using relational databases using SQL. Relational databases will be best for this application because of the type of data we will be dealing with. Most of the data FantasyMax will process will be things such as player stats, team stats, schedules, etc. The data structures will be simple and predictable, thus making relational databases an easy choice. With relational databases, we can run queries easier than we could with NoSQL. The data used by FantasyMax will, for the most part, be predictable and not see much change overtime. One issue that relational databases could cause would be scaling. Scaling with relational databases is usually done by scaling up, therefore spending more on physical hardware, RAM, etc. Our developers also already have experience with SQL, so this will make it easier to implement.

    Platform
    FantasyMax will first launch as a browser-based web application. Our development team has more experience with browser-based application development than mobile-based application development. Eventually, we realize that FantasyMax will need a mobile application. So much web traffic happens through mobile devices, so FantasyMax needs to scale to mobile devices in order to reach our maximum audience. Development for browser-based applications is pretty straight forwards as well. If we can focus on delivering the best product we can on the web, we can eventually take that knowledge and create a mobile application that handles common problems for mobile devices such as: processor power (less powerful than computers), intermittent connectivity, power management (limited battery life), and on-screen keyboards (error-prone).

    Server
    FantasyMax will utilize AWS Cloud Services. It’s really hard to go wrong with any of the big 3 cloud service providers (AWS, Microsoft Azure, or Google Cloud Services). However, we chose to use AWS for our application because they offer many tools that could help our application grow and be successful. They offer databases, analytics, security, management, and more. The services they offer set them apart for us and we decided that AWS would help maximize our application’s potential.

    Open Source
    FantasyMax will take advantage of some opensource software. As a budding business/application, we want to deliver the best product we can without spending excess money. Utilizing opensource software was an easy decision for us, allowing us to implement some key features in our application without having to create them from scratch (which can get expensive). We plan to use MySQL as our database management system. This system is opensource and will be very useful to our application. We’ve also discussed implementing some other open source features, such as a log-in page and our payment gateway.

    Development Tools
    FantasyMax has some choices that may limit our architectural choices. We plan to use a framework to create our application, and this will cause us to shift our architecture to fit the mold the framework expects. Using open-source software can also be limiting because we have no control over the evolution of opensource features we implement. Overall, I think our FantasyMax application will be simple enough where the architecture won’t be limited too much.
